# Observability and Performance Analysis

> CI Excellence does **not** include built-in APM, distributed tracing, or observability platform integration. The framework provides structured logging, GITHUB_STEP_SUMMARY reports, SARIF security reports, and a notification system. Everything beyond that requires custom implementation using the stub/hook extension points.

This section covers practical guidance for extending CI Excellence with observability and performance analysis capabilities.

---

## Pipeline Observability

### How do I instrument CI pipelines with distributed tracing (OpenTelemetry)?

CI Excellence does not include OpenTelemetry instrumentation. To add it, create a wrapper script that initializes tracing context before each pipeline stage.

**Extension approach:** Add a hook script in `ci-cd/ci-10-compile/begin-otel.sh` (and similar for other steps) that starts a span using the OpenTelemetry CLI or SDK:

```bash
#!/usr/bin/env bash
# ci-cd/ci-10-compile/begin-otel.sh
# Hook: starts an OTel span before the compile step

TRACE_ID="${TRACE_ID:-$(opentelemetry-cli generate-trace-id)}"
SPAN_ID=$(opentelemetry-cli start-span \
  --trace-id "$TRACE_ID" \
  --name "ci.compile" \
  --attribute "ci.workflow=$GITHUB_WORKFLOW" \
  --attribute "ci.run_id=$GITHUB_RUN_ID")

echo "contract:env:TRACE_ID=$TRACE_ID"
echo "contract:env:SPAN_ID=$SPAN_ID"
```

**Tools to consider:** [otel-cli](https://github.com/equinixmetal-helm/otel-cli), [go.opentelemetry.io/collector](https://opentelemetry.io/docs/collector/), or language-specific OTel SDKs. The [Honeycomb buildevents](https://github.com/honeycombio/buildevents) tool is purpose-built for CI tracing.

### How do I correlate traces across multiple workflow jobs and steps?

GitHub Actions does not propagate trace context between jobs. You must pass a trace ID explicitly.

**Practical approach:** Generate a trace ID in your first job, pass it via `GITHUB_OUTPUT`, and inject it into subsequent jobs via workflow `needs` outputs:

```yaml
jobs:
  setup:
    outputs:
      trace_id: ${{ steps.trace.outputs.trace_id }}
    steps:
      - id: trace
        run: echo "trace_id=$(uuidgen)" >> "$GITHUB_OUTPUT"
  build:
    needs: setup
    env:
      TRACE_ID: ${{ needs.setup.outputs.trace_id }}
```

Each CI script hook can then use `$TRACE_ID` to create child spans via your tracing tool.

### How do I visualize the critical path through my CI pipeline?

GitHub Actions provides built-in workflow run visualization showing job dependencies and timing. For deeper analysis, you need external tooling.

**What CI Excellence provides today:** The `GITHUB_STEP_SUMMARY` reports generated by `scripts/ci/reports/` show per-job pass/fail status tables. GitHub's workflow run UI shows a dependency graph with per-job duration.

**For deeper analysis:** Export job start/end timestamps to a tracing backend (Honeycomb, Jaeger) and visualize as a Gantt chart. Create a custom report script at `scripts/ci/reports/ci-15-timing-report.sh` that queries the GitHub API for workflow run timing:

```bash
gh api "repos/$GITHUB_REPOSITORY/actions/runs/$GITHUB_RUN_ID/timing"
```

### How do I track and alert on CI pipeline SLOs (build time, success rate, deployment frequency)?

CI Excellence does not track SLOs. GitHub provides raw data (workflow run history, timing) but no SLO framework.

**Extension approach:** Create a maintenance script (e.g., `scripts/ci/maintenance/ci-45-slo-check.sh`) that queries the GitHub API for recent workflow runs, computes metrics, and alerts via the existing notification system:

```bash
#!/usr/bin/env bash
set -euo pipefail
source "$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)/_ci-common.sh"

# Query last 100 workflow runs
RUNS=$(gh api "repos/$GITHUB_REPOSITORY/actions/runs?per_page=100&status=completed")
SUCCESS_RATE=$(echo "$RUNS" | jq '[.workflow_runs[] | select(.conclusion=="success")] | length / (.[] | length) * 100')

echo:Maint "CI Success Rate: ${SUCCESS_RATE}%"

if (( $(echo "$SUCCESS_RATE < 95" | bc -l) )); then
  echo:Error "SLO violation: success rate ${SUCCESS_RATE}% < 95%"
  # Trigger notification via existing system
fi
```

**External tools:** Sleuth, LinearB, Jellyfish, or Faros AI for DORA metrics tracking.

### How do I create custom dashboards for CI/CD metrics?

CI Excellence writes summary data to `GITHUB_STEP_SUMMARY` but does not provide dashboards. For custom dashboards, export metrics to a platform that supports them.

**Lightweight approach:** Use GitHub Actions' built-in workflow insights (Settings > Actions > Usage) for basic metrics. For custom dashboards, export data to Grafana, Datadog, or even a Google Sheet via API.

### How do I trace a deployment from commit to production?

CI Excellence report scripts log the commit SHA in release summaries (`GITHUB_SHA`). The release pipeline tracks version through prepare, build, publish, and verify stages with status logged to `GITHUB_STEP_SUMMARY`.

**What exists today:** Each report script (e.g., `scripts/ci/reports/ci-95-summary-release.sh`) logs the version and commit SHA. The post-release verification workflow checks that published artifacts match.

**For end-to-end tracing:** Add deployment event recording to your deploy scripts using GitHub's Deployments API:

```bash
gh api "repos/$GITHUB_REPOSITORY/deployments" \
  -f ref="$GITHUB_SHA" -f environment="production" -f auto_merge=false
```

### How do I implement structured logging across all CI scripts?

**This is already implemented.** CI Excellence uses the e-bash logger system via `_ci-common.sh`. All scripts use domain-tagged logging:

- `echo:Build`, `echo:Test`, `echo:Release`, `echo:Security`, etc.
- Output goes to stderr with colored prefixes: `[build]`, `[test]`, `[release]`, etc.
- Cross-cutting `[SUCCESS]` and `[ERROR]` tags for quick pass/fail grep
- Verbosity controlled by `DEBUG` env var: `DEBUG=build,test`, `DEBUG=*,-setup`, `DEBUG=*`

**Available logger tags:** `ci`, `build`, `test`, `release`, `setup`, `notify`, `maint`, `report`, `security`, `ops`, `success`, `error`

To add a custom logger tag for observability:

```bash
# In _ci-common.sh or a hook script
logger:init "perf" "${cl_yellow}${st_bold}[perf]${cl_reset} " ">&2"
```

## Performance Profiling

### How do I profile build performance to identify slow steps?

GitHub Actions shows per-step timing in the workflow run UI. For finer-grained profiling within scripts, add timing instrumentation.

**Extension approach:** Create a timing helper in a hook or directly in your customized scripts:

```bash
# Add to any CI script
STEP_START=$(date +%s%N)
# ... your build command ...
STEP_END=$(date +%s%N)
DURATION_MS=$(( (STEP_END - STEP_START) / 1000000 ))
echo:Build "Compile took ${DURATION_MS}ms"
```

Or wrap commands with `time` and capture output. For a reusable approach, add a `ci:timer` helper to `_ci-common.sh`:

```bash
ci:timer() {
  local tag="$1" label="$2"; shift 2
  local start=$(date +%s%N)
  "$@"
  local duration=$(( ($(date +%s%N) - start) / 1000000 ))
  printf:${tag^} "  %s completed in %dms\n" "$label" "$duration"
}
```

### How do I measure and optimize test execution time per test file?

This depends entirely on your test framework. CI Excellence does not instrument test execution.

**Practical guidance:** Most test runners support timing output natively:
- **Jest:** `--verbose` shows per-test timing; `jest --json` outputs structured results
- **pytest:** `--durations=10` shows the 10 slowest tests
- **Go:** `go test -v` shows per-test timing

Parse test runner output in your test scripts and append results to `GITHUB_STEP_SUMMARY`.

### How do I identify memory leaks during CI runs?

Not built in. Add memory profiling to your test scripts.

**Extension approach:** Create `scripts/ci/test/ci-15-memory-profile.sh` to run tests with memory profiling enabled:

```bash
# Node.js example
node --max-old-space-size=4096 --expose-gc node_modules/.bin/jest --logHeapUsage

# Python example
python -m memray run -o profile.bin pytest
python -m memray flamegraph profile.bin
```

### How do I profile Docker build performance layer by layer?

Not included. Use Docker's built-in profiling features.

**Practical guidance:**

```bash
# Enable BuildKit timing

# Analyze layer sizes

```

Add these commands to your Docker publish script or create a `ci-35-docker-analysis.sh` build step.

### How do I benchmark CI performance over time to detect regressions?

GitHub Actions does not track historical timing trends natively.

**Extension approach:** Record timing data as a build artifact or push it to external storage. Create `scripts/ci/reports/ci-13-timing-artifact.sh`:

```bash
{
  echo "{\"run_id\":\"$GITHUB_RUN_ID\","
  echo "\"sha\":\"$GITHUB_SHA\","
  echo "\"timestamp\":\"$(date -u +%Y-%m-%dT%H:%M:%SZ)\","
  echo "\"build_duration_s\":$BUILD_DURATION,"
  echo "\"test_duration_s\":$TEST_DURATION}"
} > timing-data.json
```

Upload as an artifact and use a scheduled workflow to aggregate historical data.

**Tools to consider:** [github-action-benchmark](https://github.com/benchmark-action/github-action-benchmark) stores benchmark results in gh-pages and provides trend charts.

### How do I use flame graphs to visualize build performance?

Not included. Flame graphs require profiling data from your specific build tools.

**Practical guidance:** For Node.js builds, use `--cpu-prof` flag. For general process profiling on Linux runners, use `perf` (if available) or `async-profiler`. Generate SVGs with [FlameGraph](https://github.com/brendangregg/FlameGraph) and upload as artifacts.

### How do I measure network I/O impact on build times?

Not built in. Network profiling requires OS-level instrumentation.

**Practical approach:** Time dependency installation separately from build steps. Compare `npm ci` or `pip install` duration across runs. Use `--prefer-offline` flags and caching to isolate network impact:

```bash
ci:timer build "dependency install" npm ci
ci:timer build "compilation" npm run build
```

### How do I profile npm/yarn/pnpm install performance?

Not included. Use package manager built-in diagnostics.

**Practical guidance:**

```bash
# npm timing
npm ci --timing 2>&1 | tee npm-timing.log

# yarn
YARN_ENABLE_TIMERS=true yarn install

# pnpm
pnpm install --reporter=ndjson > pnpm-install.log
```

Parse the output and append notable timings to `GITHUB_STEP_SUMMARY` in your setup script.

## Resource Monitoring

### How do I monitor CPU and memory usage of CI runners?

GitHub-hosted runners have fixed resources (2 vCPU, 7 GB RAM for `ubuntu-latest`). CI Excellence does not monitor runner resources.

**Extension approach:** Add resource monitoring to long-running steps:

```bash
# Snapshot resource usage during build
vmstat 5 > vmstat.log &
VMSTAT_PID=$!
npm run build
kill $VMSTAT_PID
# Upload vmstat.log as artifact
```

### How do I track disk space usage and prevent out-of-space failures?

Not built in. GitHub-hosted runners have ~14 GB free space.

**Practical approach:** Add a disk check to your setup or build scripts:

```bash
echo:Build "Disk usage before build:"
df -h / | tail -1
# After build
echo:Build "Disk usage after build:"
df -h / | tail -1
```

For large builds, free space by removing unused pre-installed tools:

```bash
sudo rm -rf /usr/share/dotnet /opt/ghc /usr/local/share/boost
```

### How do I monitor network bandwidth consumption in CI?

Not built in. Network monitoring on shared runners is limited.

**Practical approach:** Measure download sizes during dependency installation and log them. For Docker builds, track layer push/pull sizes. There is no reliable way to monitor total bandwidth on GitHub-hosted runners.

### How do I identify resource-intensive jobs that need optimization?

GitHub Actions shows per-job duration in the workflow run UI. Use the billing API for minute consumption.

**Practical approach:**

```bash
# Query workflow run timing
gh api "repos/$GITHUB_REPOSITORY/actions/runs/$GITHUB_RUN_ID/timing" | jq '.run_duration_ms'

# List most expensive workflows
gh api "repos/$GITHUB_REPOSITORY/actions/runs?per_page=100" \
  | jq '[.workflow_runs[] | {name: .name, duration: .run_started_at}]'
```

### How do I set resource limits (CPU, memory) for individual jobs?

GitHub-hosted runners have fixed resources. You cannot set per-job resource limits.

**For self-hosted runners:** Use container-based jobs with resource constraints:

```yaml
jobs:
  build:
    runs-on: self-hosted
    container:
      image: node:20
      options: --cpus 2 --memory 4g
```

### How do I monitor and optimize container resource allocation?

Not included. Container resource tuning is environment-specific.

**Practical guidance:** For Docker-based CI steps, profile resource usage locally with `docker stats`, then set appropriate limits in your workflow YAML or Docker Compose configuration.

### How do I track GitHub Actions runner queue times?

GitHub does not expose queue wait time directly. You can approximate it by comparing workflow `created_at` with the first job's `started_at`.

**Extension approach:**

```bash
RUN_DATA=$(gh api "repos/$GITHUB_REPOSITORY/actions/runs/$GITHUB_RUN_ID")
CREATED=$(echo "$RUN_DATA" | jq -r '.created_at')
STARTED=$(echo "$RUN_DATA" | jq -r '.run_started_at')
# Calculate difference for queue time estimate
```

## Real-time Monitoring

### How do I set up real-time alerts for CI failures?

**CI Excellence provides this today** via the notification system. The `scripts/ci/notification/ci-30-send-notification.sh` script uses Apprise to send alerts to Slack, Teams, Discord, Telegram, Email, and 80+ other services.

**Configuration:** Set the `APPRISE_URLS` secret with your notification endpoint:

```bash
# Slack
APPRISE_URLS=slack://T00/B00/XXXXX

# Multiple services
APPRISE_URLS="slack://T00/B00/XXX msteams://webhook discord://123/abc"
```

The notification scripts (`ci-20-determine-status.sh`, `ci-30-send-notification.sh`) automatically include repository, workflow, run URL, and trigger information.

### How do I monitor long-running workflows and set timeout alerts?

GitHub Actions supports `timeout-minutes` at the job and step level. CI Excellence does not add additional timeout monitoring.

**Practical approach:** Set timeouts in your workflow YAML:

```yaml
jobs:
  build:
    timeout-minutes: 30
    steps:
      - name: Compile
        timeout-minutes: 10
        run: ./scripts/ci/build/ci-10-compile.sh
```

For proactive alerts before timeout, add a watchdog step that runs in parallel.

### How do I create PagerDuty/OpsGenie incidents for critical CI failures?

Use the existing notification system with Apprise URLs for PagerDuty or OpsGenie.

**Configuration:**

```bash
# PagerDuty
APPRISE_URLS=pagerduty://integration_key@routing_key

# OpsGenie
APPRISE_URLS=opsgenie://api_key/
```

See the [Apprise wiki](https://github.com/caronc/apprise/wiki) for the full list of supported services.

### How do I stream CI logs to external log aggregation systems (ELK, Splunk)?

Not built in. CI Excellence logs go to stderr on the runner and are captured by GitHub Actions.

**Extension approach:** Add a log shipping step at the end of your pipeline. Create a hook or report script:

```bash
curl -k "https://splunk.example.com:8088/services/collector" \
  -H "Authorization: Splunk $SPLUNK_HEC_TOKEN" \
```

For structured log shipping, parse the e-bash logger output (which uses `[tag]` prefixes) and forward it.

### How do I implement health checks for CI infrastructure?

Not applicable for GitHub-hosted runners (GitHub manages availability). For self-hosted runners, use GitHub's runner health API.

**For self-hosted runners:**

```bash
gh api "repos/$GITHUB_REPOSITORY/actions/runners" | jq '.runners[] | {name, status, busy}'
```

### How do I monitor third-party service dependencies (NPM registry, Docker Hub)?

Not built in. Add health checks to your setup scripts.

**Practical approach:** In `scripts/ci/setup/ci-15-check-services.sh`:


### How do I track and alert on elevated error rates across all workflows?

Not built in. Use the GitHub API to query recent workflow run conclusions.

**Extension approach:** Add this to a scheduled maintenance run:

```bash
FAILURES=$(gh api "repos/$GITHUB_REPOSITORY/actions/runs?per_page=50&status=completed" \
  | jq '[.workflow_runs[] | select(.conclusion=="failure")] | length')

ERROR_RATE=$(( FAILURES * 100 / TOTAL ))

  echo:Error "Elevated failure rate: ${ERROR_RATE}%"
  # Trigger notification

```

## Trace Analysis and Debugging

### How do I capture and analyze traces from failed test runs?

CI Excellence does not capture traces. Test failure output is available in GitHub Actions logs.

**Extension approach:** Configure your test runner to output detailed failure information:

```bash
# Jest with verbose output
npx jest --verbose --no-cache 2>&1 | tee test-output.log

# Upload on failure
if [ $? -ne 0 ]; then
  # test-output.log will be uploaded as artifact by workflow
  echo:Error "Test failures detected. See test-output.log artifact."
fi
```

### How do I correlate application logs with CI pipeline logs?

CI Excellence logs use `[tag]` prefixes on stderr. Application logs go to stdout. They are interleaved in the GitHub Actions log viewer.

**Practical approach:** Tag application logs with the run ID:

```bash
export CI_RUN_ID="$GITHUB_RUN_ID"
export CI_JOB_NAME="$GITHUB_JOB"
# Application logs can include these for correlation
```

### How do I implement request tracing through microservices during E2E tests?

Not included. This requires application-level instrumentation, not CI tooling.

**Practical guidance:** Start your microservices with tracing enabled during E2E tests, generate a trace ID per test scenario, and export traces to a backend (Jaeger, Zipkin) running as a service container in your workflow:

```yaml
services:
  jaeger:
    image: jaegertracing/all-in-one:latest
    ports:
      - 16686:16686
      - 6831:6831/udp
```

### How do I debug performance issues that only occur in CI (not locally)?

CI runners differ from local machines in CPU, memory, disk I/O, and network. CI Excellence does not provide special debugging for this.

**Practical approach:**
1. Use `DEBUG=*` to enable all logger output for maximum visibility
2. Add timing instrumentation to suspect steps using `ci:timer` or `time`
3. Check runner resources with `nproc`, `free -m`, `df -h`
4. Compare with local execution using `act` (local GitHub Actions runner)
5. Use the hooks system to add diagnostic steps without modifying core scripts

### How do I capture heap dumps or core dumps from CI failures?

Not built in. Configure your runtime to generate dumps on failure.

```bash
# Node.js heap dump on OOM
node --max-old-space-size=4096 --heapsnapshot-signal=SIGUSR2 your-app.js

# Core dumps on Linux
ulimit -c unlimited
# Run your tests, then upload core file as artifact
```

### How do I trace dependency resolution performance issues?

Not built in. Use package manager diagnostics.

```bash
# npm
npm ci --timing --loglevel verbose 2>&1 | tee npm-debug.log

# pip
pip install -r requirements.txt -v 2>&1 | tee pip-debug.log
```

### How do I analyze why cache hits are low or ineffective?

GitHub Actions reports cache hit/miss in step output. CI Excellence does not add cache analytics.

**Practical approach:** Check your cache key strategy. Common issues:
- Lock file changes frequently (regenerating cache key)
- Cache key is too specific (includes volatile data)
- Cache is evicted due to 10 GB repo limit

Review cache usage via the GitHub API:

```bash
gh api "repos/$GITHUB_REPOSITORY/actions/caches" | jq '.actions_caches | sort_by(-.size_in_bytes) | .[0:10]'
```

## Performance Benchmarking

### How do I run performance benchmarks as part of CI?

Not included. Add a benchmark script to the test pipeline.

**Extension approach:** Create `scripts/ci/test/ci-25-benchmarks.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
source "$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)/_ci-common.sh"

echo:Test "Running performance benchmarks..."
hooks:do begin "${BASH_SOURCE[0]##*/}"

# Your benchmark command (hyperfine, k6, custom)
hyperfine --export-json benchmark.json './your-program'

echo:Success "Benchmarks complete"
```

Add the corresponding step to your workflow YAML.

### How do I compare benchmark results against baseline/previous commits?

Not built in. Use a benchmark tracking tool.

**Tools to consider:** [github-action-benchmark](https://github.com/benchmark-action/github-action-benchmark) stores results in gh-pages and generates comparison charts. It supports output from: `cargo bench`, `go test -bench`, `pytest-benchmark`, `hyperfine`, and custom JSON.

### How do I fail CI if performance degrades beyond threshold?

Not built in. Add threshold checks to your benchmark script.

```bash
CURRENT=$(jq '.results[0].mean' benchmark.json)
BASELINE=2.5  # seconds, from previous known-good run
THRESHOLD=1.1  # 10% regression allowed

LIMIT=$(echo "$BASELINE * $THRESHOLD" | bc)
if (( $(echo "$CURRENT > $LIMIT" | bc -l) )); then
  echo:Error "Performance regression: ${CURRENT}s > ${LIMIT}s (baseline: ${BASELINE}s)"
  exit 1
fi
```

### How do I track bundle size and prevent bloat?

Not built in. Add bundle analysis to your build pipeline.

**Extension approach:** Create `scripts/ci/build/ci-35-bundle-analysis.sh`:

```bash
# Webpack
npx webpack --json > bundle-stats.json
npx webpack-bundle-analyzer bundle-stats.json -m static -r bundle-report.html --no-open

# Check size threshold
SIZE=$(stat -c%s dist/bundle.js)
MAX_SIZE=$((500 * 1024))  # 500KB
if [ "$SIZE" -gt "$MAX_SIZE" ]; then
  echo:Error "Bundle size ${SIZE} exceeds limit ${MAX_SIZE}"
  exit 1
fi
```

**Tools to consider:** [bundlewatch](https://bundlewatch.io/), [size-limit](https://github.com/ai/size-limit), [bundlephobia](https://bundlephobia.com/).


### How do I benchmark parallel vs. sequential job execution?

GitHub Actions workflow runs are recorded with timing. Compare run durations between different workflow configurations.

**Practical approach:** Create two workflow variants (or use matrix strategy) and compare total run time via the API. This is a manual analysis task -- there is no automation for it in CI Excellence.

### How do I create performance comparison reports across branches?

**Extension approach:** Store benchmark results as artifacts keyed by branch and SHA. In PR workflows, download the main branch baseline artifact and compare:


## Cost and Resource Analytics

### How do I track cost per build/deployment?

GitHub does not provide per-run cost data via API. CI Excellence does not track costs.

**Practical approach:** Use GitHub's billing page (Settings > Billing > Actions) for aggregate data. For per-run estimates, multiply job duration by the per-minute rate for the runner type:
- Linux: $0.008/min
- Windows: $0.016/min
- macOS: $0.08/min

### How do I analyze which jobs consume the most runner minutes?

Use the GitHub API to query workflow run timing.

```bash
# Get timing for all recent runs of a specific workflow
gh api "repos/$GITHUB_REPOSITORY/actions/workflows" | jq '.workflows[] | {id, name}'

# Then for each workflow
gh api "repos/$GITHUB_REPOSITORY/actions/workflows/$WORKFLOW_ID/runs?per_page=10" \
  | jq '.workflow_runs[] | {id: .id, duration_ms: .run_started_at, conclusion: .conclusion}'
```

### How do I identify opportunities for runner consolidation?

Not included. Analyze your workflow structure manually.

**Practical guidance:** Look for:
- Jobs with minimal work that could be combined (e.g., lint + type-check)
- Sequential jobs that could be parallelized
- Jobs that could share setup via composite actions
- Duplicate setup steps across jobs (consider reusable workflows)

### How do I track cost trends over time?

Not built in. Export billing data periodically.

**Extension approach:** Create a scheduled workflow that queries the GitHub billing API and stores results:

```bash
# GitHub provides usage for the current billing cycle
gh api "orgs/$ORG/settings/billing/actions" | jq '{total_minutes_used, total_paid_minutes_used}'
```

Store results in a repository or external database for trend analysis.

### How do I allocate CI costs to teams/projects in a monorepo?

Not built in. GitHub does not provide per-workflow or per-path billing breakdowns.

**Practical approach:** Use workflow naming conventions and query the API to aggregate run minutes by workflow name. Map workflow names to teams/projects. This requires custom scripting and is not supported by CI Excellence.

### How do I optimize cost by scheduling non-urgent builds during off-peak hours?

GitHub-hosted runner pricing is flat (no off-peak discount). However, self-hosted runners or cloud-based runners may have variable pricing.

**For self-hosted runners:** Use cron-triggered workflows for non-urgent tasks (CI Excellence's maintenance pipeline already runs on a cron schedule at 2 AM UTC). Adjust timing in your workflow YAML:

```yaml
on:
  schedule:
    - cron: '0 2 * * *'  # 2 AM UTC
```

### How do I track ROI of CI performance optimizations?

Not built in. This is a manual analysis task.

**Practical approach:** Record baseline metrics (average build time, minutes consumed per week) before optimization. After changes, compare the same metrics. Calculate time saved and multiply by the per-minute runner cost and/or developer wait-time cost.

## Hard to Implement

The following capabilities are fundamentally difficult to implement within a CI pipeline framework and would require significant custom infrastructure:

- **Full distributed tracing across CI jobs** -- GitHub Actions does not support native trace context propagation between jobs. Any solution requires manually passing trace IDs through job outputs, which is fragile and verbose.

- **Real-time log streaming to external systems** -- GitHub Actions captures logs internally. Streaming requires either a sidecar process running alongside your scripts or post-hoc log shipping after the job completes. True real-time streaming is not feasible on hosted runners.

- **Per-job resource limits on GitHub-hosted runners** -- Runner resources are fixed. The only option is self-hosted runners with container-based isolation.

- **Automatic performance regression detection** -- Requires a stable baseline, statistical analysis to distinguish noise from regressions, and persistent storage for historical data. No turnkey solution exists for GitHub Actions; [github-action-benchmark](https://github.com/benchmark-action/github-action-benchmark) is the closest option.

- **Cost allocation in monorepos** -- GitHub billing is per-organization, not per-workflow or per-path. Accurate cost allocation requires custom tracking infrastructure.

- **CI infrastructure health monitoring for hosted runners** -- GitHub manages hosted runner availability. You have no visibility into or control over the underlying infrastructure. Self-hosted runners are the only option for infrastructure-level monitoring.
